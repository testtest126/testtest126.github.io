GCD ordering, mechanically · 2026

What does this print?

iOS interview concurrency puzzles aren't really puzzles — they're a mechanical procedure wearing a costume. Four rules, one method: build a chain from print to print, or admit there isn't one.

§1 · the method

Don't guess the order. Build the chain.

Every "what prints first" GCD question is really asking one thing: can you connect two print statements with an arrow that some rule forces to exist? If you can draw an unbroken chain from the first print to the last using only the rules below, that chain is the answer. If two prints have no rule linking them — no shared serial queue, no sync, no group dependency — the honest answer is "either could come first." Guessing a specific order there is the single most common way this question gets missed.

§2 · the four rules

Memorize four rules. Not four hundred examples.

Almost every ordering guarantee in a GCD snippet reduces to one of these. Learn what each one does and does not promise, and the "trick" puzzles stop being tricks.

I

A block runs top to bottom.

Inside any single closure, statements execute in the order they're written — same as a normal function body. The puzzle doesn't start until a second block enters the picture.

II

Async schedules and moves on.

Dispatching a block with .async doesn't wait for anything — it queues the block for later and immediately falls through to the next line at the call site. Code written after the dispatch call runs before that block's body starts.

III

Serial means one at a time, in order.

A serial queue — or a barrier block on a concurrent queue — finishes the block it's running before starting the next one, strictly in the order blocks arrived. Main-thread UI code is just another block on the serial main queue, which is why it obeys this same rule.

IV

Sync and wait() block the caller.

sync, DispatchGroup.wait(), and semaphore.wait() all freeze the calling thread until the awaited work finishes. group.notify(queue:) is not in this family — it hands off a block to run later and blocks nothing, which is exactly why it's easy to misread.

Three checks to run before you trust any chain you've built:

A

Deadlock check first.

Before tracing prints, ask: does a sync call target the serial queue that's already running? Or do two queues sync-wait on each other? Either one means the answer is "deadlock," not an order.

B

QoS is priority, not order.

.userInitiated versus .background changes how eagerly the system schedules a block — it is not a guarantee about which of two independent blocks prints first.

C

Concurrent ≠ parallel print order.

A concurrent queue starts blocks in arrival order, but running in parallel says nothing about which one finishes — or prints — first. No chain between two blocks means the order is genuinely unconstrained.
