Orchestration, from the inside · 2026

One answer.
A whole swarm underneath.

Ask an agent one question and you get one clean reply. Underneath, the work may have been split across a hierarchy — a root that owns the goal, orchestrators that divide it, workers that each do one small thing. Here is what that machinery looks like, and the discipline that keeps a swarm from becoming theater.

§1 · the shape

Root, orchestrator, worker — the work fans out, the answer comes back.

The canonical version has two tiers. An orchestrator (Anthropic's engineering writing calls it the lead agent) takes a goal, decomposes it into independent units, and dispatches a set of workers (subagents) — each with its own context window and tools — to run those units in parallel. The workers return condensed results; the orchestrator merges them into one answer. Anthropic names this the orchestrator-workers pattern in its guide to building effective agents.

The pattern nests. When a job is too big for one orchestrator to hold, a root sits above several orchestrators, each of which owns a branch and fans out its own workers. That is the three-tier swarm — root → orchestrators → workers — and it is exactly the shape the demo further down actually ran. That would have been pure cost with nothing to show — a swarm is spawned with work, runs it, and returns; there is no do-nothing agent to leave humming. §3 · the discipline

A swarm is pointed, not summoned.

By Anthropic's own account, its multi-agent research system beat a strong single-agent baseline on an internal research evaluation — and burned roughly fifteen times the tokens of an ordinary chat to do it. The same writeup admits the early failure mode plainly: agents that would "spawn 50 subagents for simple queries" and hunt the web for sources that were never there. The swarm is a power tool with a power tool's bill; the skill is knowing when the fan-out earns it.

It earns it when the sub-tasks are genuinely independent and the work is wide — breadth-first search, auditing many files at once, verifying a claim from several angles, a large migration. It does not earn it on a single-threaded task that one agent could carry, and it earns nothing at all when spawned with no task to hand down. The rules below are just that line, written out:

I

No task, no worker.

A worker exists to execute one concrete unit — read a file, run a search, make an edit. An agent with nothing assigned isn't a worker; it's a bill. Spawning a swarm for its own sake is the failure, not the feature.

II

Fan out for breadth, not for show.

Parallel agents pay off only when the pieces are truly independent. If unit B needs unit A's answer first, that's a sequence wearing a swarm's costume — and paying a swarm's price for nothing.

III

The root owns the result.

Merging isn't automatic. Someone above has to integrate what comes back, judge whether it meets the goal, and be accountable for the whole. A swarm with no owner returns a pile, not an answer.

IV

Every tier is a multiplier.

Cost scales with the swarm — roughly 15× a single chat is the documented order of magnitude, not a rounding error. Add a tier only when the answer it buys is worth the tokens it spends.

V

Report what you actually have.

Say plainly how many agents ran, what they cost, and which figures are measured versus illustrative. The section above does exactly that — the same rule the rest of this collection keeps circling back to.

§4 · the spine

The surface is one answer. The substance is the swarm.

This piece rhymes with the rest of the set. Each one puts something small and visible on top of something larger doing the actual work, and the two are never the same size. Trust-nothing is an agent that runs code it never read; hello-human is an agent that reads meaning that was never there; tiny-prompt is six words standing on months of invisible context — and the throughline keeps going from there. And the discipline is the same one, pointed one more way. There it was: don't trust the surface. Here it's: don't mistake the surface for the whole of what did the work — and don't summon a swarm you have no work to feed. Verify what's underneath; pay for it only when it's real.

the takeaway

The lesson of the swarm isn't spin up more agents.

It's the same one as the rest of this set: a hierarchy of agents is only worth its cost when it's pointed at real work — and the answer on the surface is never the whole of what did the work underneath.
